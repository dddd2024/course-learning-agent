一、当前状态与问题判断
1.1 当前代码的真实状态
根据当前仓库结构和你本地运行截图，项目已经具备前后端主体框架、用户登录、课程、资料、计划、测验、Agent 审计等模块；但在实际点击页面时出现了接口参数不匹配、404 Not Found、仪表盘内容空白等问题。说明项目处于“主体功能已实现，但联调和体验未完成”的阶段。

1.2 已经明确暴露的问题

1.3 当前最重要的开发原则
先稳定，再美化；先修复 P0/P1 影响演示的 bug，再追加新功能。
所有前端页面必须有 loading、empty、error 三种状态，不能出现大面积空白或原始报错。
所有后端接口必须与前端参数一致，不能一边限制 100，一边前端请求 200。
所有演示功能必须有 demo 数据支撑，不能依赖现场手工造大量数据。
所有 bug 必须能复现、能定位、能回归测试，不能只靠手工刷新确认。
二、Bug 排查总体策略
2.1 建立“可复现基线”
正式排查前，先统一运行环境和数据状态。每次修 bug 前后都按同一套命令启动、登录、访问页面，避免把数据库脏数据、端口占用、依赖缺失误判为代码 bug。
2.2 建立四类日志证据

2.3 Bug 处理闭环
复现：记录页面路径、点击步骤、请求接口、实际错误。
归类：判断是前端参数、后端路由、数据库数据、权限校验、UI 状态，还是脚本/环境。
定位：找到对应前端 API 封装、页面调用位置、后端 endpoint、schema 和测试文件。
修复：优先选择接口契约一致的修复方式，不要只在页面上吞掉错误。
回归：执行最小测试 + 手工演示路径 + 相关页面刷新。
记录：在 bug_fix_log.md 中写清楚问题、原因、改动、验证结果。
三、已暴露问题的专项修复计划
3.1 BUG-001：page_size 参数不一致
问题表现：前端请求课程列表时传入 page_size=200，后端课程接口限制 page_size <= 100，导致页面弹出校验错误。

3.2 BUG-002：Not Found 弹窗
Not Found 的根因需要通过 Network 和后端日志定位。常见原因包括：请求了不存在的课程/资料/对话 ID；页面加载时 selectedCourseId 为空；前端 API 路径与后端路由不一致；demo 数据未写入；删除后仍使用旧 ID。

3.3 BUG-003：仪表盘空白
仪表盘现在是占位页。修复方向不是“随便加几个字”，而是把它做成项目入口：展示数据概览、今日待办、最近课程、最近 Agent 运行、快速操作和系统状态。

3.4 BUG-004：启动脚本稳健化
脚本开头检查 backend/.venv 是否存在，不存在则提示创建，而不是直接失败。
检查 frontend/node_modules 是否存在，不存在则提示 npm install。
检查 8000 和 5173 端口是否被占用，给出明确提示。
所有字符串统一使用英文半角引号，保存为 UTF-8 with BOM 或稳定 UTF-8。
启动后自动打开浏览器 http://localhost:5173，可选。

六、功能成熟化提升计划
6.1 核心闭环优先级
项目最有说服力的闭环不是“能聊天”，而是：上传资料 -> 解析切块 -> 检索证据 -> Agent 回答 -> 引用追溯 -> 生成知识点 -> 生成测验 -> 记录薄弱点 -> 生成复习计划 -> Agent 审计可回放。后续功能都应服务这个闭环。

6.2 建议新增/增强的后端接口

6.3 Agent 和 RAG 质量提升
mock 模式改造：不要固定返回“梯度下降”，而应根据 retrieved_chunks 生成与材料相关的确定性回答。
引用校验增强：不仅检查 chunk_id 是否存在，还要检查 quote_text 是否真的出现在 chunk.text 中。
可靠性等级细化：高=多引用且 quote 匹配；中=单引用或弱匹配；低=仅检索相关；失败=无资料。
回答失败策略：当没有资料时，明确提示“当前课程资料中没有找到依据”，并建议上传资料或换问题。
AgentRun 记录 fallback_reason：真实模型失败、JSON 解析失败、超时、无 API Key 等。
检索可视化增强：展示 query、top_k、每个 chunk 的 score、是否被引用、来源资料、页码。
